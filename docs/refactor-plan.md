# lagom-ui: Refactor to Modern Specs

## Why

`lagom-ui` (`@fcongson/lagom-ui`, currently `0.29.0`) is a React/styled-components component library published to GitHub Packages, consumed by `lagom-starter-gatsby-decap-cms` and styled via `@fcongson/lagom-tokens`. Two things have drifted out of date at once:

1. **The token dependency moved out from under it.** `lagom-tokens` was just migrated to the W3C DTCG format on Style Dictionary v5 (see its `docs/dtcg-migration-plan.md`), bumped to `1.0.0`, but never tagged or published. `lagom-ui` still pins `^0.19.0` and has not been smoke-tested against the new build.
2. **The package itself predates several "modern spec" norms**: CommonJS-only output at an ES5 target with no `exports` map, no CI, no lint config (despite a sibling `lagom-eslint-plugin` scaffold sitting unused), two overlapping visual-regression tools, and a styling layer (styled-components) that's heavier than what the components actually need — most of them are thin wrappers around `var(--lagom-...)` references that the global stylesheet (`createTheme.ts`'s `createGlobalStyle`) already resolves.

This plan upgrades the token dependency, modernizes the package shape, and right-sizes the styling architecture, in that order — because the styling rewrite needs the new token output as its foundation, and the package-format change needs the styling rewrite settled first (CSS Modules ship real `.css` files; that changes what `files`/`exports` need to point at).

Decisions already made for this plan (confirmed with the repo owner):
- **Styling**: migrate off styled-components to zero-runtime CSS (CSS Modules), since the existing components only ever interpolate CSS custom properties — there's no real dynamic styling logic to preserve.
- **Package format**: dual ESM + CJS via `tsup`, with a proper `exports` map, to avoid breaking the existing Gatsby-based consumer.
- **Branch hygiene**: the stale `copilot` branch (and its uncommitted-to-main "WIP" docs/MDX restructuring) is being discarded, not merged. This plan starts clean from `main`.

## Current state (baseline)

- `lagom-ui` `main` is at `bc0178a` ("Fix readme"); local branches `copilot`, `chromatic`, `storybook-v8` all contain abandoned or superseded work and should be deleted.
- `package.json`: `main`/`types` fields only, no `exports` map, no `engines`, CJS-only build (`tsc -p tsconfig.prod.json`, `target: es5`), no `sideEffects` field.
- Styling: `styled-components@6.1` everywhere. Every component (`Button`, `LinkButton`, `CallToAction`, `FeaturedSection`, `Hero`, `ImageCaption`, `PageHeader`, `Quote`, `SectionHeader`, layouts `Container`/`Section`) defines a `styled.*` template that's almost entirely `var(--themed-x, var(--lagom-x))` fallback chains — no real runtime-computed styles. The one genuinely dynamic piece is `createTheme.ts`'s `createGlobalStyle`, which writes consumer theme overrides onto `:root` as custom properties.
- Token consumption: components only ever reference CSS custom properties (good — no JS-level coupling to token values). The JS variable imports (`@fcongson/lagom-tokens/js/variables/*`) are used only in Storybook docs pages (`Color.stories.tsx`, `Font.stories.tsx`, etc.), not in shipped component code.
- Testing: Jest 29 + ts-jest + Testing Library, snapshot tests per component. No Vitest.
- Storybook 8.5, addon-essentials/a11y/interactions/links, webpack5 builder.
- Visual regression: both Loki (`.loki/reference`, local) and Chromatic (`chromatic.config.json`, cloud) are configured — redundant, and only Chromatic has been used recently (`fb9033d "Try chromatic"`).
- No `.github/workflows` at all — no CI, no automated publish.
- No ESLint config in the repo, despite `lagom-eslint-plugin` existing as a sibling project (currently just an empty ESLint 9 flat-config scaffold).
- husky + lint-staged are wired up (Prettier only).
- `resolutions: { jackspeak: "2.1.1" }` is a leftover pin of unknown current necessity.
- Consumer impact: `lagom-starter-gatsby-decap-cms` depends on `@fcongson/lagom-ui@^0.29.0` and is the one real downstream smoke-test available.

## Target state

- `lagom-ui` builds with `tsup` to `dist/` (ESM `.js` + CJS `.cjs` + `.d.ts`, source maps), driven by a real `exports` map (`"."`, plus any CSS subpath exports needed for global resets), `"sideEffects"` correctly scoped to CSS imports, and `engines.node >= 20`.
- Components styled with CSS Modules (`Button.module.css` etc.) composed via `clsx` (already a dependency), referencing the same `--lagom-*`/`--themed-*` custom properties as today — visually unchanged. `styled-components` and `createGlobalStyle` are gone; theme overrides are applied by writing custom properties onto `:root` (or a scoped wrapper) via a small non-CSS-in-JS helper.
- `lagom-tokens` is tagged, published, and bumped in `lagom-ui` as the actual foundation the rest of the refactor builds on.
- One visual-regression tool (Chromatic), CI on every PR (lint, typecheck, test, build, build-storybook, Chromatic), and a publish workflow gated on a version tag.
- ESLint flat config wired in via `lagom-eslint-plugin`, run in CI and pre-commit alongside Prettier.
- Storybook upgraded off 8.5 (current major is 9.x/10.x, Vite-powered, smaller dependency tree); stale `copilot`/`chromatic`/`storybook-v8` branches deleted.
- Changesets adopted for versioning/CHANGELOG, mirroring the recommendation already made in `lagom-tokens`'s migration doc, so both repos release the same way.

## Phased plan

**Phase 0 — Branch cleanup & baseline.** Delete the `copilot`, `chromatic`, and `storybook-v8` local branches (and matching remotes if pushed) without merging — confirmed discard, not fold-in. Branch a new `refactor/modern-specs` off `main`. Build current `main` and Storybook as-is, and capture a Chromatic baseline / current `storybook-static` output — this is the regression net for the styling rewrite.

**Phase 1 — Publish `lagom-tokens`. Done.** `dtcg-migration` (plus the follow-up `outputReferences` fix, commit `4d4da39`) is merged to `main`, tagged `v1.0.0`, and published to GitHub Packages (confirmed live). Variable-name stability was verified by diffing pre/post-migration builds: all 145 `--lagom-*` names unchanged.

**Phase 2 — Bump and smoke-test the token dependency.** Resolved. Diffing a pre-migration build (`5744190`) against the DTCG migration build (`51f5290`) confirmed all 145 `--lagom-*` custom property names are unchanged. The migration did flatten four composite typography tokens (`--lagom-semantic-typography-body`, `--lagom-component-call-to-action-typography`, `--lagom-component-page-header-typography`, `--lagom-component-section-header-typography`) from `var()` reference chains to literal values; `lagom-tokens` commit `4d4da39` ("Narrow css outputReferences to per-token, not per-type") fixed this for the two tokens that didn't need it — `semantic.typography.body` and `CallToAction.typography` are back to live `var()`/`calc()` chains, confirmed by rebuilding and diffing the output. `PageHeader.typography`/`SectionHeader.typography` correctly remain flattened, since their font-size math is exponent-based (`1.5^5`/`1.5^4`) and `calc()` has no exponent operator — that's a hard constraint, not a regression. No component CSS in `lagom-ui` needs changes for this bump; it's now a non-event.

**Phase 3 — Styling migration (styled-components → plain CSS). Done.** Turned out CSS Modules (scoped/hashed class names) weren't the right mechanism: components lean on stable global BEM classes as real cross-component coupling (`FeaturedSection` selects `.lagom-container`/`.lagom-section` directly, `CallToAction` expects `.lagom-section-header` to exist, etc.). Converted every `styled.*`/`createGlobalStyle` block to a plain `.css` file (one per component/layout) imported as a side effect, keeping every class name identical. All 9 components, both layouts, `GlobalStyle`, `NoFocusOutline`, `createTheme`'s theme-override injector, `useTheme`/`ThemeContext`, the Storybook preview decorator, and the `ContentContainer` story util are converted; `styled-components` is removed from `dependencies`; `copyfiles` now ships the new `.css` files into `build/lib`; a `declare module "*.css"` shim and a jest CSS mock were added so `tsc`/`jest` don't choke on the new side-effect imports.

One thing worth knowing: a handful of styles depended on JS-computed theme values (`theme.mediaQueries.*`, `theme.sizes.maxWidthContent`) that were never actually exercised as runtime-configurable by any consumer or doc — these are now hardcoded to their current defaults (992px/768px/568px/320px breakpoints, 1456px emphasized min-width, 800px/1120px/1800px sizes). `createTheme()`'s CSS-custom-property override mechanism (button colors, fonts, header scale) is unaffected and still fully dynamic.

Also found and fixed two pre-existing bugs unrelated to this migration (confirmed via `git show HEAD` that both predate it): `Button`/`LinkButton` tests threw "Missing or invalid GrowthBookProvider" because they rendered without one — added a test-only `MockGrowthBookProvider` (`src/test-utils/growthbook-mock.tsx`, a bare `GrowthBook()` instance so no real network calls happen) and updated both test files; snapshots regenerated to drop the now-gone styled-components hash classes. And `tsc --noEmit` failed with "Cannot find namespace 'JSX'" (in this repo's own code and inside `@growthbook`/`@storybook`'s shipped `.d.ts` files) — added `"skipLibCheck": true` to `tsconfig.json`, changed the two real `JSX.Element` occurrences to `React.JSX.Element` (the `@types/react@19` form), and fixed the `build` script's `&` → `&&` so `prebuild`/`tsc`/`copyfiles` run sequentially instead of a failing `tsc` being silently swallowed. `tsc --noEmit` is clean on both configs, `jest` is all green.

**Phase 4 — Package/build modernization. Done.** Mid-implementation, switched from `tsup` to **tsdown**: tsup's own maintainers now say it's unmaintained and point to tsdown (a Rolldown-based successor from the Rolldown/Vite team) as the replacement — confirmed with the user before switching, since this was a live decision affected by info past this plan's original writing.

What changed:
- `tsc -p tsconfig.prod.json` replaced with `tsdown` (config in `tsdown.config.ts`), producing `build/lib/index.{js,mjs,d.ts}` (dual CJS+ESM) with sourcemaps, targeting ES2020.
- tsdown's built-in CSS support (`@tsdown/css`, `css: { inject: true }`) replaces the old `copyfiles` step entirely: it merges every component's `import "./Button.css"` into one `build/lib/style.css` and keeps a single `import` for it in the built JS, so `import { Button } from "@fcongson/lagom-ui"` still auto-loads styling. `copyfiles`/`rimraf` dropped from devDependencies and scripts.
- Added a real `exports` map (`.` + `./package.json`), plus legacy `main`/`module`/`types` fields for older tooling. `sideEffects: ["**/*.css"]` so consumer bundlers don't tree-shake styling away.
- `react`/`react-dom` moved from `dependencies` to `peerDependencies` (`^18.0.0 || ^19.0.0`) — the standard pattern for component libraries, avoiding a duplicate-React risk for consumers. Kept in `devDependencies` too for local tooling. `@storybook/test` moved from `dependencies` to `devDependencies` (it's Storybook-only, was never imported by shipped code).
- `engines.node: >=20.0.0` added.
- `tsconfig.json`: dropped the ES5 target and `baseUrl` (the one bare-specifier import that relied on it, in `createTheme.ts`, is now a relative import) in favor of `target: ES2020`, `module: ESNext`, `moduleResolution: bundler` — matches how tsdown/Rolldown actually resolves modules. `tsconfig.prod.json` is no longer referenced by anything (tsdown follows the real import graph from `src/index.ts`, so stories/tests are naturally excluded without needing exclude patterns) — left in place but effectively dead; candidate for removal.
- `jackspeak` resolution **removed**. It was added in commit `8225982` (2023) to pin `jackspeak@2.1.1` over `2.3.x`, at the time pulled in transitively via `rimraf`/`glob`. Since `rimraf` is dropped in this refactor (superseded by `tsdown`), that consumer is gone and nothing else in the current tree requires jackspeak — confirm via `pnpm why jackspeak` after a fresh `pnpm install`.

**Bug found during Phase 5 verification, fixed:** `sideEffects: ["**/*.css"]` only protects `.css` files from tree-shaking. `src/styles/GlobalStyle.tsx` has zero exports — it exists purely to trigger CSS side effects (`import "@fcongson/lagom-tokens/css/theme/_light.css"`, `_dark.css`, `./GlobalStyle.css`). Since it's a `.tsx` file with nothing consumed from it, webpack's `sideEffects`-based dead code elimination dropped the entire module, silently stripping the base font/color/reset CSS out of Storybook's dev build (confirmed via the bundled JS: `ThemeProvider.tsx`/`NoFocusOutline.tsx`/`resetr.css` all present, `GlobalStyle.tsx`/`GlobalStyle.css` completely absent — no build error, since tree-shaking elimination isn't an error). It's the only zero-export side-effect module in `src/` (checked every `.tsx`/`.ts` file); `NoFocusOutline.tsx` survives only because it happens to export a used class-name constant. Fixed by adding `"**/GlobalStyle.tsx"` to the `sideEffects` array. Confirmed afterward (Phase 7) that this same base CSS does end up in the published `build/lib/style.css`.

**Phase 5 — Storybook upgrade. Done.** Storybook 8.5 → 10.4, two majors. Run via Storybook's own official codemod-based upgrade tool (`npx storybook@latest upgrade`, applied twice: 8.5→9.x, then 9.x→10.x) rather than hand-editing configs, since it rewrites config/dependencies against the real installed `node_modules` and runs its own `doctor` check. `.loki/` (Loki visual-regression config) was confirmed empty and untracked, nothing to remove from version control — Chromatic was already the only visual-regression tool actually in use.

Expected breaking changes, confirmed as encountered: Storybook 9 folded `@storybook/addon-essentials`/`@storybook/addon-interactions` into core and renamed `@storybook/test`/`@storybook/theming`/`@storybook/manager-api` to `storybook/test`/`storybook/theming`/`storybook/manager-api`; Storybook 10 requires ESM-only `main.ts` (already ESM, no change needed) and Node >=20.19 or >=22.12.

**Bug found post-upgrade, fixed:** base font/color CSS wasn't applying in the running Storybook. Traced to the same `GlobalStyle.tsx` tree-shaking issue described under Phase 4 (the fix there — adding it to `sideEffects` — resolved this too, confirmed via a clean `pnpm storybook` run showing no console errors and fonts rendering correctly).

**Phase 6 — Lint & CI. Done.**

`lagom-eslint-plugin` turned out to be non-functional, not just unconfigured as originally assumed: its `package.json` points `main` at an `index.js` that doesn't exist (only one rule file lives under `lib/rules/`), and the directory isn't a git repo at all, so there's nothing to version or publish. Its one rule also only catches hardcoded hex/rgb/px values in JSX props or JS object literals, which doesn't apply post-Phase-3 - nearly everything styling-related now lives in plain `.css` files, invisible to a JS-only linter. Decided (with the user) to skip it for now and use a standard flat config instead; revisit `lagom-eslint-plugin` as separate future work if it's ever made into a real package.

What's in place:
- `eslint.config.mjs`: `@eslint/js` recommended + `typescript-eslint` recommended (non-type-checked) + `eslint-plugin-react` (`flat.recommended` + `flat['jsx-runtime']`, since React 19's automatic JSX runtime doesn't need `React` in scope) + `eslint-plugin-react-hooks` + `eslint-plugin-jsx-a11y` + `eslint-plugin-storybook` (applies its own rules only to `*.stories.*` files). `react/prop-types` turned off - confirmed nothing in `src/` uses PropTypes, everything's typed via TS (the `prop-types` dependency in `package.json` looks like dead weight from an earlier version of the codebase, not yet removed).
- ESLint pinned to `^9.39.4`, not the just-released v10 - `eslint-plugin-react`'s and `eslint-plugin-jsx-a11y`'s published `peerDependencies` both cap at `^9.7`, no `^10.0.0` yet. Jumping to ESLint 10 now would create real peer-dependency conflicts across the plugin ecosystem; staying on the latest 9.x line keeps everything compatible.
- `package.json` scripts: added `lint` (`eslint .`) and `typecheck` (`tsc --noEmit -p tsconfig.json`).
- `.lintstagedrc`: added `eslint --fix` ahead of `prettier --write` for `.ts`/`.tsx` files under `src/` and `.storybook/`.
- `.github/workflows/ci.yml`: runs on push to `main` and on PRs - install (pnpm), lint, typecheck, test, build, build-storybook. Modeled on `lagom-tokens`'s `ci.yml` but using `pnpm/action-setup` instead of yarn, and Node 20 (Storybook 10 needs >=20.19 or >=22.12).
- `.github/workflows/publish.yml`: runs on `v*` tag pushes - install, lint, typecheck, test, build, then `npm publish` to GitHub Packages using a `NODE_AUTH_TOKEN` secret, mirroring `lagom-tokens`'s `style-dictionary.yml` release workflow.

Built `eslint.config.mjs` as ESM specifically because the codebase isn't set up for `"type": "module"` (`jest.config.js` uses `module.exports`, and flipping `"type"` globally would break it) — a stray `eslint.config.js` briefly existed alongside it and had to be deleted, since ESLint's config resolution checks `.js` before `.mjs` and would otherwise have loaded the broken one first.

First real `pnpm lint` run (the first time lint had ever run on this codebase) turned up 19 real errors across multiple files — all fixed individually rather than suppressed: `configType`/unused React import cleanup in `.storybook/`, stray `!!` double-negation in `Hero.tsx`/`Section.tsx`, `Object` → `object` in a stories file, unused `imageSrc_White`/`imageSrc_Grey` vars across four story files, and dead code removed entirely (`src/themes/theme-sub.ts`, unused and `@ts-nocheck`'d). Confirmed clean on the next run.

**Phase 7 — Versioning & docs. Done for `lagom-ui`.**

README and the Storybook "About" page already said "styled with plain CSS" from Phase 3 - the original plan text above still said "CSS Modules", which was the abandoned approach, not what got built. Added one line to both about the new package shape (dual ESM+CJS, Node >= 20, `react`/`react-dom` as peer deps).

Adopted Changesets (`.changeset/config.json`, `.changeset/README.md`, `@changesets/cli` devDependency, `changeset`/`version-packages` scripts) for future releases - the actual `npm publish` to GitHub Packages still happens via the tag-triggered `publish.yml` from Phase 6, Changesets just owns version-bump proposals and `CHANGELOG.md` generation going forward. `lagom-tokens` doesn't have Changesets set up either; that's separate follow-up work in that repo, not done here.

Bumped version `0.29.0` -> `1.0.0` directly (not through Changesets, since there's no prior changeset history to consume) - hand-wrote `CHANGELOG.md`'s `1.0.0` entry summarizing Phases 1-6. This is a real breaking release: `styled-components` is gone entirely (anything importing its theme types from this package breaks), `react`/`react-dom` moved to peer dependencies, and the build output shape changed (dual ESM+CJS via tsdown instead of CJS-only via tsc).

Also removed while touching `package.json` for the bump: `prop-types` and `react-is` devDependencies - both dead weight from the styled-components era, confirmed no `PropTypes` usage anywhere in `src/`.

Confirmed clean (lint/typecheck/test) after removing `prop-types`/`react-is`. The real `tsdown` output then exposed a bug in Phase 4's `package.json` fields: `main`/`types`/`exports` pointed at `index.js`/`index.d.ts`, but tsdown (since this package has no `"type": "module"`) actually names its output `index.cjs`/`index.mjs` and splits declarations into `index.d.mts`/`index.d.cts` to avoid ambiguity - none of those old filenames exist in `build/lib/`. Published as-is, `require("@fcongson/lagom-ui")` would have 404'd. Fixed `main`/`types`/`exports` to match tsdown's actual output, with `import`/`require` conditions each getting their own `types` (`.d.mts`/`.d.cts`).

Also verified directly in the real build output: `build/lib/style.css` contains the `html, body { font-family: var(--lagom-core-font-families-body) }` rule from `GlobalStyle.css` (the Phase 4/5 tree-shaking bug fix holds in the actual published artifact, not just Storybook's dev build). `resetr.css`'s `html { font-family: sans-serif }` still comes later in the merged file and technically wins the cascade for the `html` element itself, but `GlobalStyle.css` sets `font-family` directly on `body` too (which `resetr.css` never touches), so visible text still resolves correctly via `body`'s own explicit declaration rather than inheriting from `html`. Confirmed, not a live bug - just a cascade quirk worth knowing about if this area gets touched again.

**Second bug found in the same build output, fixed:** `build/lib/index.cjs` literally started with a raw ESM `import './style.css';` statement - `require()` on it throws immediately ("Cannot use import statement outside a module"), confirmed directly (`node -e "require('./build/lib/index.cjs')"` fails). This is `@tsdown/css`'s `css.inject: true` option: per its own docs it "preserves `import` statements pointing to the emitted CSS files" - fine for the ESM output, but it does the exact same literal thing for CJS too, which isn't valid CJS syntax. Confirmed as a known, open limitation of tsdown's still-experimental CSS support (tracked at [rolldown/tsdown#472](https://github.com/rolldown/tsdown/issues/472); a maintainer confirmed CJS injection isn't supported yet as of a March 2026 discussion reply).

Fixed by turning `css.inject` back off (the tsdown default) - `build/lib/style.css` is still emitted, just no longer auto-imported by the built JS in either format. This does change the consumer-facing API: previously (styled-components) nothing needed a separate CSS import at all; now consumers need one `import "@fcongson/lagom-ui/style.css"` alongside their component imports, exposed via a new `"./style.css"` entry in `package.json`'s `exports` map. Documented in README's new "Usage" section and `CHANGELOG.md`. This is the same pattern most non-CSS-in-JS component libraries use (nothing lagom-ui-specific), and it's consistent between ESM and CJS rather than "auto-works in one, silently broken in the other."

**Third bug, found after re-running the build:** the top-level `import` statement was gone from `index.cjs`, but `node -e "require('./build/lib/index.cjs')"` now failed differently - `require()` hit a literal `require("@fcongson/lagom-tokens/css/theme/_light.css")` call partway through the bundle and choked trying to parse CSS as JS. Cause: `@fcongson/lagom-tokens` is (correctly) externalized as a real dependency rather than bundled, but that externalization carried over to its CSS imports too - tsdown left them as raw `require`/`import` calls to be resolved at runtime, which doesn't work for a `.css` file with no loader. `GlobalStyle.tsx` was the source (it imported `@fcongson/lagom-tokens/css/theme/_light.css` and `_dark.css` directly, so consumers got a theme "for free"). Checked the one real consumer (`lagom-starter-gatsby-decap-cms`): it already imports `lagom-tokens`'s theme CSS itself, directly, at its own app-layout level (`src/components/layout/index.js`) - so this was redundant there anyway, not load-bearing. Removed both imports from `GlobalStyle.tsx` entirely; consumers now pick and import their own `lagom-tokens` theme CSS, documented in README's "Usage" section alongside the `style.css` import from the previous bug fix.

Re-verified after this fix: both `require("./build/lib/index.cjs")` and `import("./build/lib/index.mjs")` resolve cleanly end-to-end, and `build/lib/style.css` contains the full merged component stylesheet (correctly, no longer `lagom-tokens`'s own theme variable *definitions* — only lagom-ui's own component CSS, since consumers now supply the token theme themselves).

**Phase 8 — Verification & publish. Done.**

Chromatic skipped by explicit decision, not oversight: `chromatic.log` showed the only historical run (March 2024, `storybook-v8` branch) failed at the "verify" step with a Playwright timeout loading the published Storybook iframe, there was no CI integration, and git history showed repeated "Try chromatic"/"Remove chromatic" cycles - there was never a working baseline to diff against. Replaced with manual visual verification of the running Storybook via browser inspection; found one pre-existing issue (a `<pre>` nested inside a `<p>` in `StyleGuide.stories.tsx`, console-warned, traced via `git blame` to a 2023-12-12 commit) that's unrelated to this refactor and was explicitly left alone.

`pnpm link` turned out to be the wrong tool for consumer verification: it doesn't resolve peer dependencies from the target's `node_modules` (pnpm's own warning), and `lagom-starter-gatsby-decap-cms` uses yarn, not pnpm, so a pnpm-side link never would have been visible to it anyway. Used `"@fcongson/lagom-ui": "file:../lagom-ui"` in the consumer's `package.json` instead, which is what actually let `yarn install && yarn build` exercise the real built package before anything was published.

That local-link build succeeded structurally on the first try (proving the API surface didn't break), but a suspiciously tiny purgecss report (`0.39 KB -> 0.39 KB, -0.00%`) gave it away: nothing in the consumer imported the new `@fcongson/lagom-ui/style.css` yet, since styled-components used to inject styles at runtime with no import needed at all - a real, expected consequence of the Phase 4 CSS-in-JS removal, not a bug. Fixed by adding the import to the consumer's `src/components/layout/index.js`, alongside its existing `lagom-tokens` theme imports.

That surfaced a second, genuinely separate bug: even with the import in place, a stale (non-`gatsby clean`) rebuild showed lagom-ui's classes landing in the wrong output chunk, and a real fresh-from-clean rebuild showed the homepage's actual linked stylesheet had *zero* `.lagom-*` classes at all. Root cause, confirmed against `gatsby-plugin-purgecss`'s own README ("Issue 5: Using npm packages with components which import css files"): it only scans this repo's own `src/**/*.{ts,js,jsx,tsx}` for class-name usage by default, so it had no way to see lagom-ui's classNames rendered from its compiled JS in `node_modules`, and stripped all of them - every time, no exceptions. This never mattered before because styled-components' runtime-injected `<style>` tags bypassed webpack's CSS/purge pipeline entirely; switching lagom-ui to plain static CSS exposed a purgecss gap in the consumer that was always latent. Fixed with a `safelist: [/^lagom-/]` entry in the consumer's `gatsby-config.js` (the plugin's own recommended fix for this exact scenario) rather than trying to get content-scanning to reach into `node_modules`. Verified afterward that the homepage's real built stylesheet contains all 29 `.lagom-*` classes.

While bumping the consumer's `@fcongson/lagom-tokens` from `^0.19.0` to `^1.0.0` alongside the lagom-ui bump (done in the same sitting, not deferred): diffed the actual generated `css/theme/_light.css` between the two versions directly (137 `--lagom-*` custom properties, byte-identical set on both sides) to confirm zero renames before bumping, rather than trusting the "no renames" claim from Phase 2's own changelog blind.

2. Both workflows referenced `secrets.TOKEN`, but the repo had zero secrets configured at all (confirmed on the Settings -> Secrets and variables -> Actions page: "This repository has no secrets"). `pnpm install` in CI failed with a 401 trying to fetch `@fcongson/lagom-tokens` from `npm.pkg.github.com`, since the committed `.npmrc`'s `${TOKEN}` placeholder never resolved to anything. Rather than have you create and rotate a manual PAT, switched both workflows to the automatic per-run `secrets.GITHUB_TOKEN` (job-level `env: TOKEN:`, plus an explicit `permissions: packages: write` block in `publish.yml`) - both repos are owned by the same account, so this only required a one-time "Manage Actions access" grant on `lagom-tokens`' own package settings page, letting `lagom-ui`'s workflow runs read it.
3. With auth fixed, the `Build` step failed on a different error entirely: `tsdown` couldn't import its own `unrun` module. Traced to `tsdown@0.22.3`'s `engines` field in `pnpm-lock.yaml`: `^22.18.0 || >=24.11.0` - a range that excludes all of Node 20.x. Both workflows were pinned to Node 20 (chosen in Phase 6 for Storybook's `>=20.19` requirement, which is compatible but not sufficient). This never surfaced locally since local machines already run newer Node. Fixed by bumping both workflows to Node 22, which satisfies both Storybook's and tsdown's requirements; `package.json`'s own `engines.node: ">=20.0.0"` for consumers was left untouched, since that's about the published runtime code, not the build tooling.

`@fcongson/lagom-ui@1.0.0` published successfully to `npm.pkg.github.com` after these fixes (confirmed live on the package page). Consumer's final state - `package.json`/`yarn.lock` pointing at the real published `^1.0.0` (not `file:`), the `style.css` import, and the purgecss safelist - committed together in `lagom-starter-gatsby-decap-cms`.

## Risks encountered (retrospective)

The plan originally called for CSS Modules and `tsup`; both changed mid-implementation (Phase 3 switched to plain global CSS with stable BEM class names, since components rely on cross-component class coupling that scoped/hashed names would have broken; Phase 4 switched to `tsdown` after tsup's own maintainers deprecated it in favor of tsdown). The token bump (Phase 2) risk closed early and cleanly: verified directly via build diff that no `--lagom-*` variable was renamed. The real risk that materialized wasn't styling or types — it was the CSS delivery mechanism itself: moving off styled-components' runtime injection to static CSS files surfaced two separate real bugs downstream, one in tsdown's own CJS/CSS handling (Phase 7) and one in the consumer's purgecss config, which had never needed to look inside `node_modules` before (Phase 8). Both are documented in place under their respective phases.

## Sequencing (as it actually happened)

Phases 0–2 (branch cleanup, `lagom-tokens` publish, token bump smoke-test) came first, in order, as planned. Phase 3 (styling rewrite) followed, then Phase 4 (package/build format), which depended on Phase 3's file shape being settled. Phases 5 and 6 (Storybook upgrade, lint/CI) ran independently of each other. Phase 7 (versioning/docs, `1.0.0` bump) and Phase 8 (consumer verification, tag, publish) closed out the migration — Chromatic was dropped from the release gate entirely (see Phase 8) in favor of manual visual verification, since there was never a working baseline to diff against.
